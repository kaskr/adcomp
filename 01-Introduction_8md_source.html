3 Introduction {#Introduction}
4 ============
5 
6 TMB (Template Model Builder) is an R package for fitting statistical latent variable models to data. It is strongly inspired by [ADMB](http://www.admb-project.org). Unlike most other R packages the model is formulated in C++. This provides great flexibility, but requires some familiarity with the C/C++ programming language.
7 
8 * TMB can calculate first and second order derivatives of the likelihood function by [AD](http://en.wikipedia.org/wiki/Automatic_differentiation), or any objective function written in C++.
9 * The objective function (and its derivatives) can be called from R. Hence, parameter estimation via e.g. `nlminb()` is easy.
10 * The user can specify that the [Laplace approximation](http://en.wikipedia.org/wiki/Laplace_approximation) should be applied to any subset of the function arguments.
11  * Yields marginal likelihood in latent variable model.
12 * Standard deviations of any parameter, or derived parameter, obtained by the ['delta method'](http://en.wikipedia.org/wiki/Delta_method).
13 * Pre and post-processing of data done in R.
14 * TMB is based on state-of-the art software: [CppAD](http://www.coin-or.org/CppAD/), [Eigen](http://eigen.tuxfamily.org/index.php?title=Main_Page), ...
15 
16 A more general introduction including the underlying theory used in TMB can be found in this [paper](https://www.jstatsoft.org/index.php/jss/article/view/v070i05/v70i05.pdf).

19 Tutorial {#Tutorial}
20 ========
21 
22 A TMB project consists of an R file (*.R) and a C++ file (*.cpp). The
23 R file does pre- and post processing of data in addition to maximizing
24 the log-likelihood contained in *.cpp. See [Examples](#Examples) for more
25 details. All R functions are documented within the standard help
26 system in R. This tutorial describes how to write the C++ file, and
27 assumes familiarity with C++ and to some extent with R.
28 
29 The purpose of the C++ program is to evaluate the objective function,
30 i.e. the negative log-likelihood of the model. The program is compiled
31 and called from R, where it can be fed to a function minimizer like
32 `nlminb()`.
33 
34 The objective function should be of the following C++ type:
35 
36 ```{.cpp}
37 #include <TMB.hpp>
38 
39 template<class Type>
40 Type objective_function<Type>::operator() ()
41 {
42 .... Here goes your C++ code .....
43 }
44 ```
45 
46 The first line includes the source code for the whole TMB package (and
47 all its dependencies). The objective function is a templated class
48 where `<Type>` is the data type of both the input values and
49 the return value of the objective function. This allows us to
50 evaluate both the objective function and its derivatives using the
51 same chunk of C++ code (via the AD package
52 [CppAD](http://www.coin-or.org/CppAD/Doc/doxydoc/html/index.html)). The
53 technical aspects of this are hidden from the user. There is however
54 one aspect that surprises the new TMB user. When a constant like "1.2"
55 is used in a calculation that affects the return value it must be
56 "cast" to Type:
57 
58 ```{.cpp}
59 Type nll; // Define variable that holds the return value (neg. log. lik)
60 nll = Type(1.2); // Assign value 1.2; a cast is needed.
61 ```
62 
63 ## Obtaining data and parameter values from R
64 Obviously, we will need to pass both data and parameter values to the
65 objective function. This is done through a set of macros that TMB
66 defines for us.
67 
68 ### List of data macros
69 
70 > `DATA_ARRAY()`, `DATA_FACTOR()`, `DATA_IARRAY()`, `DATA_IMATRIX()`, `DATA_INTEGER()`, `DATA_IVECTOR()`, `DATA_MATRIX()`, `DATA_SCALAR()`, `DATA_SPARSE_MATRIX()`, `DATA_STRING()`, `DATA_STRUCT()`, `DATA_UPDATE()`, `DATA_VECTOR()`
71 
72 ### List of parameter macros
73 
74 > `PARAMETER()`, `PARAMETER_ARRAY()`, `PARAMETER_MATRIX()`, `PARAMETER_VECTOR()`
75 
76 To see which macros are available start typing
77 `DATA_` or `PARAMETER_` in the Doxygen search field of
78 your browser (you may need to refresh the browser window between each
79 time you make a new search). A simple example if you want to read a
80 vector of numbers (doubles) is the following
81 
82 ```{.cpp}
83 DATA_VECTOR(x); // Vector x(0),x(1),...,x(n-1), where n is the length of x
84 ```
85 
86 Note that all vectors and matrices in TMB uses a **zero-based**
87 indexing scheme. It is not necessary to explicitly pass the dimension
88 of `x`, as it can be retrieved inside the C++ program:
89 
90 ```{.cpp}
91 int n = x.size();
92 ```
93 
94 ## An extended C++ language
95 TMB extends C++ with functionality that is important for formulating
96 likelihood functions. You have different toolboxes available:
97 
98 - Standard C++ used for infrastructure like loops etc.
99 - Vector, matrix and array library (see [Matrices and arrays](#matrix_arrays))
100 - Probability distributions (see [Densities](#Densities) and [R style distributions](#R_style_distribution))
101 
102 In addition to the variables defined through the `DATA_` or
103 `PARAMETER_` macros there can be "local" variables, for which
104 ordinary C++ scoping rules apply. There must also be a variable that
105 holds the return value (neg. log. likelihood).
106 
107 ```{.cpp}
108 DATA_VECTOR(x); // Vector x(0), x(1), ..., x(n-1)
109 Type tmp = x(1);
110 Type nll = tmp * tmp;
111 ```
112 
113 As in ordinary C++ local variable tmp must be assigned a value before
114 it can enter into a calculation.
115 
116 ## Statistical modelling
117 TMB can handle complex statistical problems with hierarchical
118 structure (latent random variables) and multiple data sources. Latent
119 random variables must be continuous (discrete distributions are not
120 handled). The `PARAMETER_` macros are used to pass two types
121 of parameters.
122 
123 - **Parameters**: to be estimated by maximum likelihood. These include
124  fixed effects and variance components in the mixed model
125  literature. They will also correspond to hyper parameters with
126  non-informative priors in the Bayesian literature.
127 - **Latent random variables**: to be integrated out of the likelihood
128  using a Laplace approximation.
129 
130 Which of these are chosen is controlled from R, via the
131 `random` argument to the function `MakeADFun`. However,
132 on the C++ side it is usually necessary to assign a probability
133 distribution to the parameter.
134 
135 The purpose of the C++ program is to calculate the (negative) joint
136 density of data and latent random variables. Each datum and individual
137 latent random effect gives a contribution to log likelihood, which may
138 be though of as a "distribution assignment" by users familiar with
139 software in the BUGS family.
140 
141 ```{.cpp}
142 PARAMETER_VECTOR(u); // Latent random variable
143 Type nll = Type(0); // Return value
144 nll -= dnorm(u(0),0,1,true) // Distributional assignment: u(0) ~ N(0,1)
145 ```
146 
147 The following rules apply:
148 
149 - Distribution assignments do not need to take place before the latent
150  variable is used in a calculation.
151 - More complicated distributional assignments are allowed, say
152  *u(0)-u(1) ~ N(0,1)*, but this requires the user to have a deeper
153  understanding of the probabilistic aspects of the model.
154 - For latent variables only normal distributions should be used
155  (otherwise the Laplace approximation will perform poorly). For
156  response variables all probability distributions (discrete or
157  continuous) are allowed. If a non-gaussian latent is needed the
158  "transformation trick" can be used.
159 - The namespaces [R style distributions](#R_style_distribution) and [Densities](#Densities) contain
160  many probability distributions, including multivariate normal
161  distributions. For probability distributions not available from
162  these libraries, the user can use raw C++ code:
163 
164  ```{.cpp}
165  DATA_VECTOR(y); // Data vector
166  Type nll = Type(0); // Return value
167  nll -= sum(-log(Type(1.0)+y*y)); // y are i.i.d. Cauchy distributed
168  ```
169 
170 See [Toolbox](#Toolbox) for more about statistical modelling.

187 Matrices and arrays {#matrix_arrays}
188 ===================
189 
190 ## Relationship to R
191 In R you can apply both matrix multiplication (`%*%`) and elementwise
192 multiplication (`*`) to objects of type "matrix", i.e. it is the
193 operator that determines the operation. In TMB we instead have two
194 different types of objects, while the multiplication operator (`*`) is
195 the same:
196 
197 - `matrix`: linear algebra
198 - `array`: elementwise operations; () and [] style indexing.
199 - `vector`: can be used in linear algebra with `matrix`, but at the
200  same time admits R style element-wise operations.
201 
202 See the file \ref matrix_arrays.cpp for examples of use.
203 
204 ## Relationship to Eigen
205 The TMB types `matrix` and `array` (in dimension 2) inherits
206 directly from the the Eigen types Matrix and Array. The advanced user
207 of TMB will benefit from familiarity with the
208 [Eigen documentation](http://eigen.tuxfamily.org/dox/group__TutorialMatrixClass.html).
209 Note that arrays of dimension 3 or higher are specially implemented in
210 TMB, i.e. are not directly inherited from Eigen.

213 R style probability distributions {#R_style_distribution}
214 =================================
215 
216 Attempts have been made to make the interface (function name and
217 arguments) as close as possible to that of R.
218 
219 - The densities (`d...`) are provided both in the discrete and
220  continuous case, cumulative distributions (`p...`) and inverse
221  cumulative distributions (`q...`) are provided only for continuous
222  distributions.
223 - Scalar and `vector` arguments (in combination) are supported, but
224  not `array` or `matrix` arguments.
225 - The last argument (of type `int`) corresponds to the `log`
226  argument in R: 1=logaritm, 0=ordinary scale. `true` (logaritm) and
227  `false` (ordinary scale) can also be used.
228 - Vector arguments must all be of the same length (no recycling of
229  elements). If vectors of different lengths are used an "out of
230  range" error will occur, which can be picked up by the debugger.
231 - `DATA_IVECTOR()` and `DATA_INTEGER()` cannot be used with
232  probability distributions, except possibly for the last (log)
233  argument.
234 - An example:
235 
236  ```{.cpp}
237  DATA_SCALAR(y);
238  DATA_VECTOR(x);
239  vector<Type> rate(10);
240  matrix<Type> rate_matrix(10, 10);
241  dexp(y, rate, true); // OK, returns vector of length 10 of log-densities
242  dexp(x, rate, true); // OK if x is length 10
243  dexp(x, rate_matrix, true); // Error, matrix arguments not allowed
244  ```
245 - To sum over elements in the vector returned use
246 
247  ```{.cpp}
248  sum(dexp(x,rate));
249  ```

252 Multivariate distributions {#Densities}
253 ==========================
254 
255 The namespace
256 
257 ```{.cpp}
258 using namespace density;
259 ```
260 
261 gives access to a variety of multivariate normal distributions:
262 
263 - Multivariate normal distributions specified via a covariance matrix
264  (structured or unstructured).
265 - Autoregressive (AR) processes.
266 - Gaussian Markov random fields (GMRF) defined on regular grids or
267  defined via a (sparse) precision matrix.
268 - Separable covariance functions, i.e. time-space separability.
269 
270 These seemingly unrelated concepts are all implemented via the notion
271 of a `distribution`, which explains why they are placed in the same
272 namespace. You can combine two `distributions`, and this lets you
273 build up complex multivariate distributions using extremely compact
274 notation. Due to the flexibility of the approach it is more abstract
275 than other parts of TMB, but here it will be explained from
276 scratch. Before looking at the different categories of multivariate
277 distributions we note the following which is of practical importance:
278 
279 - All members in the `density` namespace return the **negative** log
280  density, opposed to the univariate densities in [R style distributions](#R_style_distribution).
281 
282 ## Multivariate normal distributions
283 Consider a zero-mean multivariate normal distribution
284 with covariance matrix *Sigma* (symmetric positive definite),
285 that we want to evaluate at _x_:
286 
287 ```{.cpp}
288 int n = 10;
289 vector<Type> x(n); // Evaluation point
290 x.fill(0.0); // Point of evaluation: x = (0,0,...,0)
291 ```
292 
293 The negative log-normal density is evaluated as follows:
294 
295 ```{.cpp}
296 using namespace density;
297 matrix<Type> Sigma(n,n); // Covariance matrix
298 // ..... User must assign value to Sigma here
299 res = MVNORM(Sigma)(x); // Evaluate negative log likelihod
300 ```
301 
302 In the last line `MVNORM(Sigma)` should be interpreted as a
303 multivariate density, which via the last parenthesis `(x)` is
304 evaluated at `x`. A less compact way of expressing this is
305 
306 ```{.cpp}
307 MVNORM_t<Type> your_dmnorm(Sigma);
308 res = your_dmnorm(x);
309 ```
310 
311 in which `your_dmnorm` is a variable that holds the "density".
312 
313 Note, that the latter way (using the `MVNORM_t`) is _more efficient_
314 when you need to evaluate the density more than once, i.e. for different values of `x`.
315 
316 _Sigma_ can be parameterized in different ways. Due to the symmetry of
317 *Sigma* there are at most *n(n+1)/2* free parameters (*n* variances
318 and *n(n-1)/2* correlation parameters). If you want to estimate all of
319 these freely (modulo the positive definite constraint) you can use
320 `UNSTRUCTURED_CORR()` to specify the correlation matrix, and
321 `VECSCALE()` to specify variances. `UNSTRUCTURED_CORR()` takes as
322 input a vector a dummy parameters that internally is used to build the
323 correlation matrix via its cholesky factor.
324 
325 ```{.cpp}
326 using namespace density;
327 int n = 10;
328 vector<Type> unconstrained_params(n*(n-1)/2); // Dummy parameterization of correlation matrix
329 vector<Type> sds(n); // Standard deviations
330 res = VECSCALE(UNSTRUCTURED_CORR(unconstrained_params),sds)(x);
331 ```
332 
333 If all elements of `dummy_params` are estimated we are in effect
334 estimating a full correlation matrix without any constraints on its
335 elements (except for the mandatory positive definiteness). The actual
336 value of the correlation matrix, but not the full covariance matrix,
337 can easily be assessed using the `.cov()` operator
338 
339 ```{.cpp}
340 matrix<Type> Sigma(n,n);
341 Sigma = UNSTRUCTURED_CORR(unconstrained_params).cov();
342 REPORT(Sigma); // Report back to R session
343 ```
344 
345 ## Autoregressive processes
346 Consider a stationary univariate Gaussian AR1 process
347 *x(t),t=0,...,n-1*. The stationary distribution is choosen so that:
348 
349 - *x(t)* has mean 0 and variance 1 (for all *t*).
350 
351 The multivariate density of the vector *x* can be evaluated as follows
352 
353 ```{.cpp}
354 int n = 10;
355 using namespace density;
356 
357 vector<Type> x(n); // Evaluation point
358 x.fill(0.0); // Point of evaluation: x = (0,0,...,0)
359 Type rho = 0.2; // Correlation parameter
360 res = AR1(rho)(x); // Evaluate negative log-density of AR1 process at point x
361 ```
362 
363 Due to the assumed stationarity the correlation parameter must
364 satisfy:
365 
366 - **Stationarity** constraint: -1 < *rho* < 1
367 
368 Note that *cor[x(t),x(t-1)] = rho*.
369 
370 The `SCALE()` function can be used to set the standard deviation.
371 
372 ```{.cpp}
373 Type sigma = 2.1; // standard deviation of x
374 res = SCALE(AR1(rho),sigma)(x);
375 ```
376 
377 Now, *var[x(t)] = sigma^2*. Because all elements of `x` are scaled by
378 the same constant we use SCALE rather than VECSCALE.
379 
380 #### Multivariate AR1 processes
381 This is the first real illustration of how distributions can be used
382 as building blocks to obtain more complex distributions. Consider the
383 *p* dimensional AR1 process
384 
385 ```{.cpp}
386 int n = 10; // Number of time steps
387 int p=3; // dim(x)
388 array<Type> x(p,n); // Evaluation point
389 ```
390 
391 The columns in `x` refer to the different time points. We then
392 evaluate the (negative log) joint density of the time series.
393 
394 ```{.cpp}
395 MVNORM_t<Type> your_dmnorm(Sigma); // Density of x(t)
396 Type phi; // Correlation parameter
397 res = AR1(phi,your_dmnorm)(x);
398 ```
399 
400 Note the following:
401 
402 - We have introduced an intermediate variable `your_dmnorm`, which
403  holds the p-dim density marginal density of *x(t)*. This is a
404  zero-mean normal density with covariance matrix `Sigma`.
405 - All *p* univarite time series have the same serial correlation *phi*.
406 - The multivariate process *x(t)* is stationary in the same sense as
407  the univariate AR1 process described above.
408 
409 #### Higher order AR processes
410 
411 There also exists \ref ARk_t of arbitrary autoregressive order.
412 
413 ## Gaussian Markov random fields (GMRF)
414 GMRF may be defined in two ways:
415 
416 1. Via a (sparse) precision matrix Q.
417 2. Via a d-dimensional lattice.
418 
419 For further details please see \ref GMRF_t. Under 1) a sparse Q
420 corresponding to a Matern covariance function can be obtained via the
421 \ref R_inla namespace.
422 
423 ## Separable construction of covariance (precision) matrices
424 A typical use of separability is to create space-time models with a
425 sparse precision matrix. Details are given in \ref SEPARABLE_t. Here
426 we give a simple example.
427 
428 Assume that we study a quantity `x` that changes both in space and
429 time. For simplicity we consider only a one-dimensional space. We
430 discretize space and time using equidistant grids, and assume that the
431 distance between grid points is 1 in both dimensions. We then define
432 an `AR1(rho_s)` process in space and one in time `AR1(rho_t)`. The
433 separable assumption is that two points `x1` and `x2`, separated in
434 space by a distance `ds` and in time by a distance `dt`, have
435 correlation given by
436 
437 `rho_s^ds*rho_t^dt`
438 
439 This is implemented as
440 
441 ```{.cpp}
442 using namespace density;
443 int n_s = 10; // Number of grid points in space
444 int n_t = 10; // Number of grid points in time
445 Type rho_s = 0.2; // Correlation in space
446 Type rho_t = 0.4; // Correlation in time
447 
448 array<Type> x(n_s,n_t);
449 x.setZero(); // x = 0
450 
451 res = SEPARABLE(AR1(rho_t),AR1(rho_s))(x);
452 ```
453 
454 Note that the arguments to `SEPARABLE()` are given in the opposite order
455 to the dimensions of `x`.

458 Example collection {#Examples}
459 ==================
460 
461 - A list of all examples is found on the "Examples" tab on the top of the page.
462 - Locations of example files: `adcomp/tmb_examples` and `adcomp/TMB/inst/examples`.
463 - For each example there is both a `.cpp` **and** a `.R` file. Take for instance the linear regression example:
464 - **C++ template**
465 
466  ```{cpp}
467  // Simple linear regression.
468  #include <TMB.hpp>
469  template<class Type>
470  Type objective_function<Type>::operator() ()
471  {
472  DATA_VECTOR(Y);
473  DATA_VECTOR(x);
474  PARAMETER(a);
475  PARAMETER(b);
476  PARAMETER(logSigma);
477  ADREPORT(exp(2*logSigma));
478  Type nll = -sum(dnorm(Y, a+b*x, exp(logSigma), true));
479  return nll;
480  }
481 
482  ```
483 
484 - **Controlling R code**
485 
486  ```r
487  library(TMB)
488  compile("linreg.cpp")
489  dyn.load(dynlib("linreg"))
490  set.seed(123)
491  data <- list(Y = rnorm(10) + 1:10, x=1:10)
492  parameters <- list(a=0, b=0, logSigma=0)
493  obj <- MakeADFun(data, parameters, DLL="linreg")
494  obj$hessian <- TRUE
495  opt <- do.call("optim", obj)
496  opt
497  opt$hessian ## <-- FD hessian from optim
498  obj$he() ## <-- Analytical hessian
499  sdreport(obj)
500  ```
501 
502 - To run this example use the R command
503 
504  ```r
505  source("linreg.R")
506  ```

555 Compilation and run time errors {#Errors}
556 ===============================
557 
558 The R interface to the debugger (gdb) is documented as part of the R
559 help system, i.e. you can type `?gdbsource` in R to get info. The
560 current document only adresses isses that the relate to C++.
561 
562 ## Compilation errors
563 It may be hard to understand the compilation errors for the following
564 reasons
565 
566 - The Eigen libraries use templated C++ which generate non-intuitive
567  error messages.
568 
569 ## Run time errors
570 Run time errors are broadly speaking of two types:
571 
572 - Out-of-bounds (you are "walking out of an array")
573 - Floating point exceptions
574 
575 You can use the debugger to locate both types of errors, but the
576 procedure is a little bit different in the two cases. The following
577 assumes that you have the GNU debugger `gdb` installed.
578 
579 ### Out-of-bounds error
580 An example is:
581 
582 ```{.cpp}
583 vector<Type> y(4);
584 y(5); // 5 is not a valid index value here
585 ```
586 
587 This will cause TMB and R to crash with the following error message:
588 
589 > TMB has received an error from Eigen. The following condition was not met:
590 > index >= 0 && index < size()
591 > Please check your matrix-vector bounds etc., or run your program through a debugger.
592 > Aborted (core dumped)
593 
594 So, you must restart R and give the commands
595 
596 
597 ```r
598 library(TMB)
599 gdbsource("my_project.R")
600 ```
601 
602 > \#5 objective_function<double>::operator() (this=<optimized out>) at nan_error_ex.cpp:11
603 
604 and you can see that the debugger points to line number 11 in the .cpp
605 file. `gdbsource()` is an R function that is part of TMB.
606 
607 ### Floating point exception
608 If you on the other hand perform an illegal mathematical operation,
609 such as
610 
611 ```{.cpp}
612 Type f = sqrt(-1.);
613 ```
614 
615 R will not crash, but the objective function will return a NaN value.
616 However, you will not know in which part of your C++ code the error
617 occured. By including the `fenv.h` library (part of many C++
618 compilers, but can otherwise be downloaded from
619 http://www.scs.stanford.edu/histar/src/uinc/fenv.h)
620 
621 `nan_error_ex.cpp`:
622 
623 ```{cpp}
624 // Illustrates how to make the debugger catch a floating point error.
625 #include <TMB.hpp>
626 #include <fenv.h> // Extra line needed
627 
628 template<class Type>
629 Type objective_function<Type>::operator() ()
630 {
631  feenableexcept(FE_INVALID | FE_OVERFLOW | FE_DIVBYZERO | FE_UNDERFLOW); // Extra line needed
632 
633  DATA_SCALAR(lambda);
634  PARAMETER(x);
635  Type f;
636  f = sqrt(-1.); // FE_INVALID ( sqrt(-1.) returns NaN )
637  //f = 1./0.; // FE_DIVBYZERO ( division by zero )
638  //f = exp(100000.); // FE_OVERFLOW ( exp(100000.) returns Inf ) [Does not work on all platforms]
639  //f = exp(-100000.); // FE_UNDERFLOW ( exp(-100000.) returns 0 )
640  return f;
641 }
642 ```
643 
644 
645 a floating point exception will be turned into an actual error that
646 can be picked up by the debugger. There are only two extra lines that
647 need to be included ("//Extra line needed" in the above example).
648 
649 When we try to run this program in the usual way, the program crashes:
650 
651 
652 ```r
653 source("nan_error_ex.R")
654 ```
655 
656 > Floating point exception (core dumped)
657 > tmp3>
658 
659 At this stage you should run the debugger to find out that the
660 floating point exception occurs at line number 14:
661 
662 
663 ```r
664 library(TMB)
665 gdbsource("nan_error_ex.R")
666 ```
667 
668 > \#1 0x00007ffff0e7eb09 in objective_function<double>::operator() (this=<optimized out>) at nan_error_ex.cpp:14
669 
670 This enabling of floating point errors applies to R as well as the TMB program.
671 For more elaborate R-scripts it may therefore happen that a NaN occurs in the R-script
672 before the floating point exception in the TMB program (i.e. the problem of interest) happens.
673 To circumvent this problem one can run without NaN debugging enabled and
674  save the parameter vector that gave the floating point exception (e.g. `badpar <- obj$env$last.par` after the NaN evaluation),
675 then enable NaN debugging, re-compile, and evaluate `obj$env$f( badpar, type="double")`.
676 
677 ### Missing casts for vectorized functions
678 
679 TMB vectorized functions cannot be called directly with expressions, for example the following will fail to compile:
680 
681 ```{.cpp}
682 DATA_VECTOR(x);
683 // Don't do this! Doesn't compile
684 vector<Type> out = lgamma(x + 1);
685 ```
686 
687 > error: could not convert ‘atomic::D_lgamma(const CppAD::vector<Type>&) ...
688 > from ‘double’ to ‘Eigen::CwiseBinaryOp<Eigen::internal::scalar_sum_op<double, double>, ... >’
689 
690 Eigen lazy-evaluates expressions, and the templating of lgamma means we expect to return a "x + y"-typed object, which it obviously can't do.
691 
692 To work around this, cast the input:
693 
694 ```{.cpp}
695 DATA_VECTOR(x);
696 vector<Type> out = lgamma(vector<Type>(x + 1));
697 ```

700 Toolbox {#Toolbox}
701 =======
702 
703 First read the *Statistical Modelling* section of [Tutorial](#Tutorial).
704 
705 ### Non-normal latent variables (random effects)
706 
707 The underlying latent random variables in TMB must be Gaussian for the
708 Laplace approximation to be accurate. To obtain other distributions,
709 say a gamma distribution, the "transformation trick" can be used. We
710 start out with normally distributed variables `u` and transform these
711 into new variables `w` via the `pnorm` and `qgamma` functions as
712 follows:
713 
714 ```{.cpp}
715 PARAMETER_VECTOR(u); // Underlying latent random variables
716 Type nll=Type(0.0);
717 nll -= sum(dnorm(u,Type(0),Type(1),true)); // Assign N(0,1) distribution u
718 vector<Type> v = pnorm(u,Type(0),Type(1)); // Uniformly distributed variables (on [0,1])
719 vector<Type> w = qgamma(v,shape,scale);
720 ```
721 
722 `w` now has a gamma distribution.
723 
724 ### Discrete latent variables
725 
726 The Laplace approximation can not be applied to discrete latent
727 variables that occur in mixture models and HMMs (Hidden Markov
728 models). However, such likelihoods have analytic expressions, and may
729 be coded up in TMB. TMB would still calculate the exact gradient of
730 the HMM likelihood.
731 
732 ### Mixture models
733 
734 Although mixture models are a special case of discrete latent variable
735 models, they do deserve special attention. Consider the case that we
736 want a mixture of two zero-mean normal distributions (with different
737 standard deviations). This can be implemented as:
738 
739 ```{.cpp}
740 DATA_VECTOR(x);
741 PARAMETER_VECTOR(sigma); // sigma0 og sigma1
742 PARAMETER(p); // Mixture proportion of model 0
743 Type nll=Type(0.0);
744 nll -= sum( log( p * dnorm(x, Type(0), sigma(0), false)
745  + (1.0-p) * dnorm(x, Type(0), sigma(1), false) ) );
746 ```
747 
748 ### Time series
749 Autoregressive (AR) processes may be implemented using the compact
750 notation of section [Densities](#Densities). The resulting AR process may be
751 applied both in the observational part and in the distribution of a
752 latent variable.
753 
754 Nonlinear time must be implemented from scratch, as in the example
755 \ref thetalog.cpp
756 
757 ### Spatial models
758 TMB has strong support for spatial model and space-time models via the
759 `GMRF()` and `SEPARABLE()` functions, and the notion of a
760 `distribution`. The reader is referred to section [Densities](#Densities) for
761 details and examples.

764 C++ tutorial {#CppTutorial}
765 ============
766 
767 ### I know R but not C++
768 
769 Summary of how syntax differs between R and C++:
770 
771 ```{.cpp}
772  R code C++/TMB code
773 
774 Comments # // // Comment symbol
775 Constants 3.4 Type(3.4); // Explicit casting recommended in TMB
776 Scalar x = 5.2 Type x = Type(5.2); // Variables must have type
777 Arrays x = numeric(10) vector<Type> x(10); // C++ code here does NOT initialize to 0
778 Indexing x[1]+x[10] x(0)+x(9); // C++ indexing is zero-based
779 Loops for(i in 1:10) for(int i=1;i<=10;i++) // Integer i must be declared in C++
780 Increments x[1] = x[1] + 3 x(0) += 3.0; // += -= *= /= incremental operators in C++
781 ```
782 
783 It is important to note the following difference compared to R:
784 
785 > Vectors, matrices and arrays are **not** zero-initialized in C++.
786 
787 A zero initialized object is created using Eigens `setZero()`:
788 
789 ```{.cpp}
790 matrix<Type> m(4,5);
791 m.setZero();
792 ```
793 
794 ### I know C++
795 TMB specific C++ include:
796 - You should not use `if(x)` statements where `x` is a `PARAMETER`,
797  or is derived from a variable of type `PARAMETER`. (It is OK to use
798  `if` on `DATA` types, however.) TMB will remove the `if(x)`
799  statement, so the code will produce unexpected results.
