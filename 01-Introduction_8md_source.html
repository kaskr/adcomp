3 Introduction {#Introduction}
4 ============
5 
6 TMB (Template Model Builder) is an R package for fitting statistical latent variable models to data. It is strongly inspired by [ADMB](http://www.admb-project.org). Unlike most other R packages the model is formulated in C++. This provides great flexibility, but requires some familiarity with the C/C++ programming language.
7 
8 * TMB can calculate first and second order derivatives of the likelihood function by [AD](http://en.wikipedia.org/wiki/Automatic_differentiation), or any objective function written in C++.
9 * The objective function (and its derivatives) can be called from R. Hence, parameter estimation via e.g. `nlminb()` is easy.
10 * The user can specify that the [Laplace approximation](http://en.wikipedia.org/wiki/Laplace_approximation) should be applied to any subset of the function arguments.
11  * Yields marginal likelihood in latent variable model.
12 * Standard deviations of any parameter, or derived parameter, obtained by the ['delta method'](http://en.wikipedia.org/wiki/Delta_method).
13 * Pre and post-processing of data done in R.
14 * TMB is based on state-of-the art software: [CppAD](http://www.coin-or.org/CppAD/), [Eigen](http://eigen.tuxfamily.org/index.php?title=Main_Page), ...
15 
16 A more general introduction including the underlying theory used in TMB can be found in this [paper](https://www.jstatsoft.org/index.php/jss/article/view/v070i05/v70i05.pdf).

19 Tutorial {#Tutorial}
20 ========
21 
22 A TMB project consists of an R file (*.R) and a C++ file (*.cpp). The
23 R file does pre- and post processing of data in addition to maximizing
24 the log-likelihood contained in *.cpp. See [Examples](#Examples) for more
25 details. All R functions are documented within the standard help
26 system in R. This tutorial describes how to write the C++ file, and
27 assumes familiarity with C++ and to some extent with R.
28 
29 The purpose of the C++ program is to evaluate the objective function,
30 i.e. the negative log-likelihood of the model. The program is compiled
31 and called from R, where it can be fed to a function minimizer like
32 `nlminb()`.
33 
34 The objective function should be of the following C++ type:
35 
36 ```{.cpp}
37 #include <TMB.hpp>
38 
39 template<class Type>
40 Type objective_function<Type>::operator() ()
41 {
42 .... Here goes your C++ code .....
43 }
44 ```
45 
46 The first line includes the source code for the whole TMB package (and
47 all its dependencies). The objective function is a templated class
48 where `<Type>` is the data type of both the input values and
49 the return value of the objective function. This allows us to
50 evaluate both the objective function and its derivatives using the
51 same chunk of C++ code (via the AD package
52 [CppAD](http://www.coin-or.org/CppAD/Doc/doxydoc/html/index.html)). The
53 technical aspects of this are hidden from the user. There is however
54 one aspect that surprises the new TMB user. When a constant like "1.2"
55 is used in a calculation that affects the return value it must be
56 "cast" to Type:
57 
58 ```{.cpp}
59 Type nll; // Define variable that holds the return value (neg. log. lik)
60 nll = Type(1.2); // Assign value 1.2; a cast is needed.
61 ```
62 
63 ## Obtaining data and parameter values from R
64 Obviously, we will need to pass both data and parameter values to the
65 objective function. This is done through a set of macros that TMB
66 defines for us.
67 
68 ### List of data macros
69 
70 > `DATA_ARRAY()`, `DATA_FACTOR()`, `DATA_IARRAY()`, `DATA_IMATRIX()`, `DATA_INTEGER()`, `DATA_IVECTOR()`, `DATA_MATRIX()`, `DATA_SCALAR()`, `DATA_SPARSE_MATRIX()`, `DATA_STRING()`, `DATA_STRUCT()`, `DATA_UPDATE()`, `DATA_VECTOR()`
71 
72 ### List of parameter macros
73 
74 > `PARAMETER()`, `PARAMETER_ARRAY()`, `PARAMETER_MATRIX()`, `PARAMETER_VECTOR()`
75 
76 To see which macros are available start typing
77 `DATA_` or `PARAMETER_` in the Doxygen search field of
78 your browser (you may need to refresh the browser window between each
79 time you make a new search). A simple example if you want to read a
80 vector of numbers (doubles) is the following
81 
82 ```{.cpp}
83 DATA_VECTOR(x); // Vector x(0),x(1),...,x(n-1), where n is the length of x
84 ```
85 
86 Note that all vectors and matrices in TMB uses a **zero-based**
87 indexing scheme. It is not necessary to explicitly pass the dimension
88 of `x`, as it can be retrieved inside the C++ program:
89 
90 ```{.cpp}
91 int n = x.size();
92 ```
93 
94 ## An extended C++ language
95 TMB extends C++ with functionality that is important for formulating
96 likelihood functions. You have different toolboxes available:
97 
98 - Standard C++ used for infrastructure like loops etc.
99 - Vector, matrix and array library (see [Matrices and arrays](#matrix_arrays))
100 - Probability distributions (see [Densities](#Densities) and [R style distributions](#R_style_distribution))
101 
102 In addition to the variables defined through the `DATA_` or
103 `PARAMETER_` macros there can be "local" variables, for which
104 ordinary C++ scoping rules apply. There must also be a variable that
105 holds the return value (neg. log. likelihood).
106 
107 ```{.cpp}
108 DATA_VECTOR(x); // Vector x(0), x(1), ..., x(n-1)
109 Type tmp = x(1);
110 Type nll = tmp * tmp;
111 ```
112 
113 As in ordinary C++ local variable tmp must be assigned a value before
114 it can enter into a calculation.
115 
116 ## Statistical modelling
117 TMB can handle complex statistical problems with hierarchical
118 structure (latent random variables) and multiple data sources. Latent
119 random variables must be continuous (discrete distributions are not
120 handled). The `PARAMETER_` macros are used to pass two types
121 of parameters.
122 
123 - **Parameters**: to be estimated by maximum likelihood. These include
124  fixed effects and variance components in the mixed model
125  literature. They will also correspond to hyper parameters with
126  non-informative priors in the Bayesian literature.
127 - **Latent random variables**: to be integrated out of the likelihood
128  using a Laplace approximation.
129 
130 Which of these are chosen is controlled from R, via the
131 `random` argument to the function `MakeADFun`. However,
132 on the C++ side it is usually necessary to assign a probability
133 distribution to the parameter.
134 
135 The purpose of the C++ program is to calculate the (negative) joint
136 density of data and latent random variables. Each datum and individual
137 latent random effect gives a contribution to log likelihood, which may
138 be though of as a "distribution assignment" by users familiar with
139 software in the BUGS family.
140 
141 ```{.cpp}
142 PARAMETER_VECTOR(u); // Latent random variable
143 Type nll = Type(0); // Return value
144 nll -= dnorm(u(0),0,1,true) // Distributional assignment: u(0) ~ N(0,1)
145 ```
146 
147 The following rules apply:
148 
149 - Distribution assignments do not need to take place before the latent
150  variable is used in a calculation.
151 - More complicated distributional assignments are allowed, say
152  *u(0)-u(1) ~ N(0,1)*, but this requires the user to have a deeper
153  understanding of the probabilistic aspects of the model.
154 - For latent variables only normal distributions should be used
155  (otherwise the Laplace approximation will perform poorly). For
156  response variables all probability distributions (discrete or
157  continuous) are allowed. If a non-gaussian latent is needed the
158  "transformation trick" can be used.
159 - The namespaces [R style distributions](#R_style_distribution) and [Densities](#Densities) contain
160  many probability distributions, including multivariate normal
161  distributions. For probability distributions not available from
162  these libraries, the user can use raw C++ code:
163  ```{.cpp}
164  DATA_VECTOR(y); // Data vector
165  Type nll = Type(0); // Return value
166  nll -= sum(-log(Type(1.0)+y*y)); // y are i.i.d. Cauchy distributed
167  ```
168 
169 See [Toolbox](#Toolbox) for more about statistical modelling.

186 Matrices and arrays {#matrix_arrays}
187 ===================
188 
189 ## Relationship to R
190 In R you can apply both matrix multiplication (`%*%`) and elementwise
191 multiplication (`*`) to objects of type "matrix", i.e. it is the
192 operator that determines the operation. In TMB we instead have two
193 different types of objects, while the multiplication operator (`*`) is
194 the same:
195 
196 - `matrix`: linear algebra
197 - `array`: elementwise operations; () and [] style indexing.
198 - `vector`: can be used in linear algebra with `matrix`, but at the
199  same time admits R style element-wise operations.
200 
201 See the file \ref matrix_arrays.cpp for examples of use.
202 
203 ## Relationship to Eigen
204 The TMB types `matrix` and `array` (in dimension 2) inherits
205 directly from the the Eigen types Matrix and Array. The advanced user
206 of TMB will benefit from familiarity with the
207 [Eigen documentation](http://eigen.tuxfamily.org/dox/group__TutorialMatrixClass.html).
208 Note that arrays of dimension 3 or higher are specially implemented in
209 TMB, i.e. are not directly inherited from Eigen.

212 R style probability distributions {#R_style_distribution}
213 =================================
214 
215 Attempts have been made to make the interface (function name and
216 arguments) as close as possible to that of R.
217 
218 - The densities (`d...`) are provided both in the discrete and
219  continuous case, cumulative distributions (`p...`) and inverse
220  cumulative distributions (`q...`) are provided only for continuous
221  distributions.
222 - Scalar and `vector` arguments (in combination) are supported, but
223  not `array` or `matrix` arguments.
224 - The last argument (of type `int`) corresponds to the `log`
225  argument in R: 1=logaritm, 0=ordinary scale. `true` (logaritm) and
226  `false` (ordinary scale) can also be used.
227 - Vector arguments must all be of the same length (no recycling of
228  elements). If vectors of different lengths are used an "out of
229  range" error will occur, which can be picked up by the debugger.
230 - `DATA_IVECTOR()` and `DATA_INTEGER()` cannot be used with
231  probability distributions, except possibly for the last (log)
232  argument.
233 - An example:
234  ```{.cpp}
235  DATA_SCALAR(y);
236  DATA_VECTOR(x);
237  vector<Type> rate(10);
238  matrix<Type> rate_matrix(10, 10);
239  dexp(y, rate, true); // OK, returns vector of length 10 of log-densities
240  dexp(x, rate, true); // OK if x is length 10
241  dexp(x, rate_matrix, true); // Error, matrix arguments not allowed
242  ```
243 - To sum over elements in the vector returned use
244  ```{.cpp}
245  sum(dexp(x,rate));
246  ```

249 Multivariate distributions {#Densities}
250 ==========================
251 
252 The namespace
253 
254 ```{.cpp}
255 using namespace density;
256 ```
257 
258 gives access to a variety of multivariate normal distributions:
259 
260 - Multivariate normal distributions specified via a covariance matrix
261  (structured or unstructured).
262 - Autoregressive (AR) processes.
263 - Gaussian Markov random fields (GMRF) defined on regular grids or
264  defined via a (sparse) precision matrix.
265 - Separable covariance functions, i.e. time-space separability.
266 
267 These seemingly unrelated concepts are all implemented via the notion
268 of a `distribution`, which explains why they are placed in the same
269 namespace. You can combine two `distributions`, and this lets you
270 build up complex multivariate distributions using extremely compact
271 notation. Due to the flexibility of the approach it is more abstract
272 than other parts of TMB, but here it will be explained from
273 scratch. Before looking at the different categories of multivariate
274 distributions we note the following which is of practical importance:
275 
276 - All members in the `density` namespace return the **negative** log
277  density, opposed to the univariate densities in [R style distributions](#R_style_distribution).
278 
279 ## Multivariate normal distributions
280 Consider a zero-mean multivariate normal distribution
281 with covariance matrix *Sigma* (symmetric positive definite),
282 that we want to evaluate at _x_:
283 
284 ```{.cpp}
285 int n = 10;
286 vector<Type> x(n); // Evaluation point
287 x.fill(0.0); // Point of evaluation: x = (0,0,...,0)
288 ```
289 
290 The negative log-normal density is evaluated as follows:
291 
292 ```{.cpp}
293 using namespace density;
294 matrix<Type> Sigma(n,n); // Covariance matrix
295 // ..... User must assign value to Sigma here
296 res = MVNORM(Sigma)(x); // Evaluate negative log likelihod
297 ```
298 
299 In the last line `MVNORM(Sigma)` should be interpreted as a
300 multivariate density, which via the last parenthesis `(x)` is
301 evaluated at `x`. A less compact way of expressing this is
302 
303 ```{.cpp}
304 MVNORM_t<Type> your_dmnorm(Sigma);
305 res = your_dmnorm(x);
306 ```
307 
308 in which `your_dmnorm` is a variable that holds the "density".
309 
310 Note, that the latter way (using the `MVNORM_t`) is _more efficient_
311 when you need to evaluate the density more than once, i.e. for different values of `x`.
312 
313 _Sigma_ can be parameterized in different ways. Due to the symmetry of
314 *Sigma* there are at most *n(n+1)/2* free parameters (*n* variances
315 and *n(n-1)/2* correlation parameters). If you want to estimate all of
316 these freely (modulo the positive definite constraint) you can use
317 `UNSTRUCTURED_CORR()` to specify the correlation matrix, and
318 `VECSCALE()` to specify variances. `UNSTRUCTURED_CORR()` takes as
319 input a vector a dummy parameters that internally is used to build the
320 correlation matrix via its cholesky factor.
321 
322 ```{.cpp}
323 using namespace density;
324 int n = 10;
325 vector<Type> unconstrained_params(n*(n-1)/2); // Dummy parameterization of correlation matrix
326 vector<Type> sds(n); // Standard deviations
327 res = VECSCALE(UNSTRUCTURED_CORR(unconstrained_params),sds)(x);
328 ```
329 
330 If all elements of `dummy_params` are estimated we are in effect
331 estimating a full correlation matrix without any constraints on its
332 elements (except for the mandatory positive definiteness). The actual
333 value of the correlation matrix, but not the full covariance matrix,
334 can easily be assessed using the `.cov()` operator
335 
336 ```{.cpp}
337 matrix<Type> Sigma(n,n);
338 Sigma = UNSTRUCTURED_CORR(unconstrained_params).cov();
339 REPORT(Sigma); // Report back to R session
340 ```
341 
342 ## Autoregressive processes
343 Consider a stationary univariate Gaussian AR1 process
344 *x(t),t=0,...,n-1*. The stationary distribution is choosen so that:
345 
346 - *x(t)* has mean 0 and variance 1 (for all *t*).
347 
348 The multivariate density of the vector *x* can be evaluated as follows
349 
350 ```{.cpp}
351 int n = 10;
352 using namespace density;
353 
354 vector<Type> x(n); // Evaluation point
355 x.fill(0.0); // Point of evaluation: x = (0,0,...,0)
356 Type rho = 0.2; // Correlation parameter
357 res = AR1(rho)(x); // Evaluate negative log-density of AR1 process at point x
358 ```
359 
360 Due to the assumed stationarity the correlation parameter must
361 satisfy:
362 
363 - **Stationarity** constraint: -1 < *rho* < 1
364 
365 Note that *cor[x(t),x(t-1)] = rho*.
366 
367 The `SCALE()` function can be used to set the standard deviation.
368 
369 ```{.cpp}
370 Type sigma = 2.1; // standard deviation of x
371 res = SCALE(AR1(rho),sigma)(x);
372 ```
373 
374 Now, *var[x(t)] = sigma^2*. Because all elements of `x` are scaled by
375 the same constant we use SCALE rather than VECSCALE.
376 
377 #### Multivariate AR1 processes
378 This is the first real illustration of how distributions can be used
379 as building blocks to obtain more complex distributions. Consider the
380 *p* dimensional AR1 process
381 
382 ```{.cpp}
383 int n = 10; // Number of time steps
384 int p=3; // dim(x)
385 array<Type> x(p,n); // Evaluation point
386 ```
387 
388 The columns in `x` refer to the different time points. We then
389 evaluate the (negative log) joint density of the time series.
390 
391 ```{.cpp}
392 MVNORM_t<Type> your_dmnorm(Sigma); // Density of x(t)
393 Type phi; // Correlation parameter
394 res = AR1(phi,your_dmnorm)(x);
395 ```
396 
397 Note the following:
398 
399 - We have introduced an intermediate variable `your_dmnorm`, which
400  holds the p-dim density marginal density of *x(t)*. This is a
401  zero-mean normal density with covariance matrix `Sigma`.
402 - All *p* univarite time series have the same serial correlation *phi*.
403 - The multivariate process *x(t)* is stationary in the same sense as
404  the univariate AR1 process described above.
405 
406 #### Higher order AR processes
407 
408 There also exists \ref ARk_t of arbitrary autoregressive order.
409 
410 ## Gaussian Markov random fields (GMRF)
411 GMRF may be defined in two ways:
412 
413 1. Via a (sparse) precision matrix Q.
414 2. Via a d-dimensional lattice.
415 
416 For further details please see \ref GMRF_t. Under 1) a sparse Q
417 corresponding to a Matern covariance function can be obtained via the
418 \ref R_inla namespace.
419 
420 ## Separable construction of covariance (precision) matrices
421 A typical use of separability is to create space-time models with a
422 sparse precision matrix. Details are given in \ref SEPARABLE_t. Here
423 we give a simple example.
424 
425 Assume that we study a quantity `x` that changes both in space and
426 time. For simplicity we consider only a one-dimensional space. We
427 discretize space and time using equidistant grids, and assume that the
428 distance between grid points is 1 in both dimensions. We then define
429 an `AR1(rho_s)` process in space and one in time `AR1(rho_t)`. The
430 separable assumption is that two points `x1` and `x2`, separated in
431 space by a distance `ds` and in time by a distance `dt`, have
432 correlation given by
433 
434 `rho_s^ds*rho_t^dt`
435 
436 This is implemented as
437 
438 ```{.cpp}
439 using namespace density;
440 int n_s = 10; // Number of grid points in space
441 int n_t = 10; // Number of grid points in time
442 Type rho_s = 0.2; // Correlation in space
443 Type rho_t = 0.4; // Correlation in time
444 
445 array<Type> x(n_s,n_t);
446 x.setZero(); // x = 0
447 
448 res = SEPARABLE(AR1(rho_t),AR1(rho_s))(x);
449 ```
450 
451 Note that the arguments to `SEPARABLE()` are given in the opposite order
452 to the dimensions of `x`.

455 Example collection {#Examples}
456 ==================
457 
458 - A list of all examples is found on the "Examples" tab on the top of the page.
459 - Locations of example files: `adcomp/tmb_examples` and `adcomp/TMB/inst/examples`.
460 - For each example there is both a `.cpp` **and** a `.R` file. Take for instance the linear regression example:
461 - **C++ template**
462 
463  ```{cpp}
464  // Simple linear regression.
465  #include <TMB.hpp>
466  template<class Type>
467  Type objective_function<Type>::operator() ()
468  {
469  DATA_VECTOR(Y);
470  DATA_VECTOR(x);
471  PARAMETER(a);
472  PARAMETER(b);
473  PARAMETER(logSigma);
474  ADREPORT(exp(2*logSigma));
475  Type nll = -sum(dnorm(Y, a+b*x, exp(logSigma), true));
476  return nll;
477  }
478 
479  ```
480 
481 - **Controlling R code**
482 
483  ```r
484  library(TMB)
485  compile("linreg.cpp")
486  dyn.load(dynlib("linreg"))
487  set.seed(123)
488  data <- list(Y = rnorm(10) + 1:10, x=1:10)
489  parameters <- list(a=0, b=0, logSigma=0)
490  obj <- MakeADFun(data, parameters, DLL="linreg")
491  obj$hessian <- TRUE
492  opt <- do.call("optim", obj)
493  opt
494  opt$hessian ## <-- FD hessian from optim
495  obj$he() ## <-- Analytical hessian
496  sdreport(obj)
497  ```
498 
499 - To run this example use the R command
500 
501  ```r
502  source("linreg.R")
503  ```

551 Compilation and run time errors {#Errors}
552 ===============================
553 
554 The R interface to the debugger (gdb) is documented as part of the R
555 help system, i.e. you can type `?gdbsource` in R to get info. The
556 current document only adresses isses that the relate to C++.
557 
558 ## Compilation errors
559 It may be hard to understand the compilation errors for the following
560 reasons
561 
562 - The Eigen libraries use templated C++ which generate non-intuitive
563  error messages.
564 
565 ## Run time errors
566 Run time errors are broadly speaking of two types:
567 
568 - Out-of-bounds (you are "walking out of an array")
569 - Floating point exceptions
570 
571 You can use the debugger to locate both types of errors, but the
572 procedure is a little bit different in the two cases. The following
573 assumes that you have the GNU debugger `gdb` installed.
574 
575 ### Out-of-bounds error
576 An example is:
577 
578 ```{.cpp}
579 vector<Type> y(4);
580 y(5); // 5 is not a valid index value here
581 ```
582 
583 This will cause TMB and R to crash with the following error message:
584 
585 > TMB has received an error from Eigen. The following condition was not met:
586 > index >= 0 && index < size()
587 > Please check your matrix-vector bounds etc., or run your program through a debugger.
588 > Aborted (core dumped)
589 
590 So, you must restart R and give the commands
591 
592 
593 ```r
594 library(TMB)
595 gdbsource("my_project.R")
596 ```

603 ### Floating point exception
604 If you on the other hand perform an illegal mathematical operation,
605 such as
606 
607 ```{.cpp}
608 Type f = sqrt(-1.);
609 ```
610 
611 R will not crash, but the objective function will return a NaN value.
612 However, you will not know in which part of your C++ code the error
613 occured. By including the `fenv.h` library (part of many C++
614 compilers, but can otherwise be downloaded from
615 http://www.scs.stanford.edu/histar/src/uinc/fenv.h)
616 
617 `nan_error_ex.cpp`:
618 
619 ```{cpp}
620 // Illustrates how to make the debugger catch a floating point error.
621 #include <TMB.hpp>
622 #include <fenv.h> // Extra line needed
623 
624 template<class Type>
625 Type objective_function<Type>::operator() ()
626 {
627  feenableexcept(FE_INVALID | FE_OVERFLOW | FE_DIVBYZERO | FE_UNDERFLOW); // Extra line needed
628 
629  DATA_SCALAR(lambda);
630  PARAMETER(x);
631  Type f;
632  f = sqrt(-1.); // FE_INVALID ( sqrt(-1.) returns NaN )
633  //f = 1./0.; // FE_DIVBYZERO ( division by zero )
634  //f = exp(100000.); // FE_OVERFLOW ( exp(100000.) returns Inf ) [Does not work on all platforms]
635  //f = exp(-100000.); // FE_UNDERFLOW ( exp(-100000.) returns 0 )
636  return f;
637 }
638 ```
639 
640 
641 a floating point exception will be turned into an actual error that
642 can be picked up by the debugger. There are only two extra lines that
643 need to be included ("//Extra line needed" in the above example).
644 
645 When we try to run this program in the usual way, the program crashes:
646 
647 
648 ```r
649 source("nan_error_ex.R")
650 ```
651 
652 > Floating point exception (core dumped)
653 > tmp3>
654 
655 At this stage you should run the debugger to find out that the
656 floating point exception occurs at line number 14:
657 
658 
659 ```r
660 library(TMB)
661 gdbsource("nan_error_ex.R")
662 ```
663 
664 > \#1 0x00007ffff0e7eb09 in objective_function<double>::operator() (this=<optimized out>) at nan_error_ex.cpp:14
665 
666 This enabling of floating point errors applies to R as well as the TMB program.
667 For more elaborate R-scripts it may therefore happen that a NaN occurs in the R-script
668 before the floating point exception in the TMB program (i.e. the problem of interest) happens.
669 To circumvent this problem one can run without NaN debugging enabled and
670  save the parameter vector that gave the floating point exception (e.g. `badpar <- obj$env$last.par` after the NaN evaluation),
671 then enable NaN debugging, re-compile, and evaluate `obj$env$f( badpar, type="double")`.
672 
673 ### Missing casts for vectorized functions
674 
675 TMB vectorized functions cannot be called directly with expressions, for example the following will fail to compile:
676 
677 ```{.cpp}
678 DATA_VECTOR(x);
679 // Don't do this! Doesn't compile
680 vector<Type> out = lgamma(x + 1);
681 ```
682 
683 > error: could not convert atomic::D_lgamma(const CppAD::vector<Type>&) ...
684 > from double to Eigen::CwiseBinaryOp<Eigen::internal::scalar_sum_op<double, double>, ... >
685 
686 Eigen lazy-evaluates expressions, and the templating of lgamma means we expect to return a "x + y"-typed object, which it obviously can't do.
687 
688 To work around this, cast the input:
689 
690 ```{.cpp}
691 DATA_VECTOR(x);
692 vector<Type> out = lgamma(vector<Type>(x + 1));
693 ```

696 Toolbox {#Toolbox}
697 =======
698 
699 First read the *Statistical Modelling* section of [Tutorial](#Tutorial).
700 
701 ### Non-normal latent variables (random effects)
702 
703 The underlying latent random variables in TMB must be Gaussian for the
704 Laplace approximation to be accurate. To obtain other distributions,
705 say a gamma distribution, the "transformation trick" can be used. We
706 start out with normally distributed variables `u` and transform these
707 into new variables `w` via the `pnorm` and `qgamma` functions as
708 follows:
709 
710 ```{.cpp}
711 PARAMETER_VECTOR(u); // Underlying latent random variables
712 Type nll=Type(0.0);
713 nll -= sum(dnorm(u,Type(0),Type(1),true)); // Assign N(0,1) distribution u
714 vector<Type> v = pnorm(u,Type(0),Type(1)); // Uniformly distributed variables (on [0,1])
715 vector<Type> w = qgamma(v,shape,scale);
716 ```
717 
718 `w` now has a gamma distribution.
719 
720 ### Discrete latent variables
721 
722 The Laplace approximation can not be applied to discrete latent
723 variables that occur in mixture models and HMMs (Hidden Markov
724 models). However, such likelihoods have analytic expressions, and may
725 be coded up in TMB. TMB would still calculate the exact gradient of
726 the HMM likelihood.
727 
728 ### Mixture models
729 
730 Although mixture models are a special case of discrete latent variable
731 models, they do deserve special attention. Consider the case that we
732 want a mixture of two zero-mean normal distributions (with different
733 standard deviations). This can be implemented as:
734 
735 ```{.cpp}
736 DATA_VECTOR(x);
737 PARAMETER_VECTOR(sigma); // sigma0 og sigma1
738 PARAMETER(p); // Mixture proportion of model 0
739 Type nll=Type(0.0);
740 nll -= sum( log( p * dnorm(x, Type(0), sigma(0), false)
741  + (1.0-p) * dnorm(x, Type(0), sigma(1), false) ) );
742 ```
743 
744 ### Time series
745 Autoregressive (AR) processes may be implemented using the compact
746 notation of section [Densities](#Densities). The resulting AR process may be
747 applied both in the observational part and in the distribution of a
748 latent variable.
749 
750 Nonlinear time must be implemented from scratch, as in the example
751 \ref thetalog.cpp
752 
753 ### Spatial models
754 TMB has strong support for spatial model and space-time models via the
755 `GMRF()` and `SEPARABLE()` functions, and the notion of a
756 `distribution`. The reader is referred to section [Densities](#Densities) for
757 details and examples.

760 C++ tutorial {#CppTutorial}
761 ============
762 
763 ### I know R but not C++
764 
765 Summary of how syntax differs between R and C++:
766 
767 ```{.cpp}
768  R code C++/TMB code
769 
770 Comments # // // Comment symbol
771 Constants 3.4 Type(3.4); // Explicit casting recommended in TMB
772 Scalar x = 5.2 Type x = Type(5.2); // Variables must have type
773 Arrays x = numeric(10) vector<Type> x(10); // C++ code here does NOT initialize to 0
774 Indexing x[1]+x[10] x(0)+x(9); // C++ indexing is zero-based
775 Loops for(i in 1:10) for(int i=1;i<=10;i++) // Integer i must be declared in C++
776 Increments x[1] = x[1] + 3 x(0) += 3.0; // += -= *= /= incremental operators in C++
777 ```
778 
779 It is important to note the following difference compared to R:
780 
781 > Vectors, matrices and arrays are **not** zero-initialized in C++.
782 
783 A zero initialized object is created using Eigens `setZero()`:
784 
785 ```{.cpp}
786 matrix<Type> m(4,5);
787 m.setZero();
788 ```
789 
790 ### I know C++
791 TMB specific C++ include:
792 - You should not use `if(x)` statements where `x` is a `PARAMETER`,
793  or is derived from a variable of type `PARAMETER`. (It is OK to use
794  `if` on `DATA` types, however.) TMB will remove the `if(x)`
795  statement, so the code will produce unexpected results.
