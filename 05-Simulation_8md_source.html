3 Simulation {#Simulation}
4 ==========
5 
6 When building models in TMB it is generally recommended to test the
7 implementation on simulated data. Obviously, data can be simulated
8 from R and passed to the C++ template. In practice this amounts to
9 implementing the model twice and is thus a strong way to validate the
10 implementation of the model. However, with increased model complexity
11 it becomes inconvenient to maintain two separate
12 implementations. Therefore, TMB allows the the user to write the
13 simulation code as an integrated part of the C++ model template.
14 
15 
16 ## Overview of simulation methods in TMB
17 
18 ### Standard generators
19 
20 The TMB simulation routines use the same naming convention as the R
21 simulators. For instance `rnorm()` is used to simulate from a normal
22 distribution. However, the argument convention is slightly
23 different:
24 
25 1. `rnorm(n, mu, sd)` draws `n` simulations from a normal
26  distribution. Unlike R this works for **scalar parameters only**.
27 2. `rnorm(mu, sd)` is a TMB specific variant that works for
28  **mixed scalar and vector input**. Output length follows the length of the
29  longest input (no re-cycling) hence is consistent with `dnorm(mu, sd)`.
30 
31 Currently the following simulators are implemented:
32 
33 > `rnorm()`, `rpois()`, `runif()`, `rbinom()`, `rgamma()`, `rexp()`, `rbeta()`, `rf()`, `rlogis()`, `rt()`, `rweibull()`, `rcompois()`, `rtweedie()`, `rnbinom()`, `rnbinom2()`
34 
35 ### Generators for density objects
36 
37 Objects from the density namespace have their own `simulate()`
38 method. Taking the multivariate normal distribution as example we have
39 the following ways to draw a simulation:
40 
41 1. `MVNORM(Sigma).simulate()` returns a vector with a simulation from
42  the multivariate normal distribution. The void argument version is
43  only available when there is no ambiguity in the dimension of the
44  output. In the `MVNORM` case the dimension of the output is known
45  from the dimension of `Sigma`. In other cases e.g. `AR1(phi)` the
46  dimension of the output is not known hence the void argument
47  version is not available.
48 2. `MVNORM(Sigma).simulate(x)` pass `x` by reference and writes the
49  simulation directly to `x` without returning anything. This version
50  is available for *all* the classes because the dimension of the
51  simulation can always be deduced from `x`.
52 
53 ### Controlling the random seed
54 
55 All TMB simulation methods are based on R's random number
56 generator. It follows that the random seed can be controlled from R
57 the usual way using `set.seed` even though the simulation is performed
58 on the C++ side.
59 
60 ## Simulation blocks
61 
62 Simulation functions can be called from anywhere in the C++
63 program. However, usually one should put the simulation code inside
64 specialized *simulation blocks* that allows the code to only be
65 executed when requested from R.
66 
67 ### A linear regression example
68 
69 A complete example extending the example \ref linreg.cpp
70 with simulation code is:
71 
72 
73 
74 
75 ```{cpp}
76 
77 #include <TMB.hpp>
78 template<class Type>
79 Type objective_function<Type>::operator() ()
80 {
81  DATA_VECTOR(y);
82  DATA_VECTOR(x);
83  PARAMETER(a);
84  PARAMETER(b);
85  PARAMETER(sd);
86  vector<Type> mu = a + b * x;
87  Type nll = -sum(dnorm(y, mu, sd, true));
88  SIMULATE {
89  y = rnorm(mu, sd); // Simulate response
90  REPORT(y); // Report the simulation
91  }
92  return nll;
93 }
94 
95 ```
96 
97 
98 > The `SIMULATE` block marks the simulation and is not executed by default.
99 
100 
101 
102 We compile the C++-file and the model object is constructed as usual:
103 
104 
105 ```r
106 obj <- MakeADFun(data, parameters, DLL="linreg")
107 ```
108 
109 Now a simulation can be generated with
110 
111 
112 ```r
113 set.seed(1) ## optional
114 obj$simulate()
115 ```

120 ## [7] 0.4874291 0.7383247 0.5757814 -0.3053884
121 ```
122

124 data. A complete dataset can be generated by:

144 Here we did not explicitely state the parameter values to use with the
145 simulation. The `simulate` method takes an additional argument `par`
146 that can be used for this.
147 
148 > The default parameter values used for the simulation is
149 > `obj$env$last.par`.

151 ### A simulation study
152 
153 Simulating datasets from known parameters and re-estimationg those
154 parameters can be done generically by:
155 
156 
157 ```r
158 sim <- replicate(50, {
159  simdata <- obj$simulate(par=obj$par, complete=TRUE)
160  obj2 <- MakeADFun(simdata, parameters, DLL="linreg", silent=TRUE)
161  nlminb(obj2$par, obj2$fn, obj2$gr)$par
162 })
163 ```

191 ### Advanced examples
192 
193 The examples \ref sam.cpp and \ref ar1_4D.cpp
194 includes more advanced simulation code. The latter demonstrates how to
195 simulate from the density objects:
196 
197 
198 ```{cpp}
199 // Separable covariance on 4D lattice with AR1 structure in each direction.
200 #include <TMB.hpp>
201 
202 /* Parameter transform */
203 template <class Type>
204 Type f(Type x){return Type(2)/(Type(1) + exp(-Type(2) * x)) - Type(1);}
205 
206 template<class Type>
207 Type objective_function<Type>::operator() ()
208 {
209  DATA_VECTOR(N)
210  PARAMETER_ARRAY(eta);
211  PARAMETER(transf_phi); /* fastest running dim */
212  Type phi=f(transf_phi);
213  ADREPORT(phi);
214 
215  using namespace density;
216  Type res=0;
217 
218  res+=AR1(phi,AR1(phi,AR1(phi,AR1(phi))))(eta);
219 
220  // logdpois = N log lam - lam
221  for(int i=0;i<N.size();i++)res-=N[i]*eta[i]-exp(eta[i]);
222 
223  SIMULATE {
224  AR1(phi,AR1(phi,AR1(phi,AR1(phi)))).simulate(eta);
225  vector<Type> lam = exp(eta);
226  N = rpois(lam);
227  REPORT(eta);
228  REPORT(N);
229  }
230 
231  return res;
232 
233 }
234 ```
235 
236 
237 In this example the 4D-array `eta` is passed to the simulator by
238 reference. Thereby the simulator knows the dimension of `eta` and can
239 fill `eta` with a simulation.
240 
241 ### Further notes
242 
243 The above example only used one simulation block. In general there is
244 no limitation on the number of simulation blocks that can be used in a
245 model and simulation blocks can use temporaries calculated outside the
246 blocks (as demonstrated in the linear regression example). For clarity
247 reasons, it is often a good idea to add a simulation block after each
248 likelihood contribution. However, note that simulation blocks are in
249 general **not** commutative (unlike likelihood accumulation). It is
250 therefore further recommended to add likelihood contributions of
251 random effects in the natural hierarchical order.
